# QnA - 12/04/2026

## Godot Rust (gdext)

**Q: Slime sight Area2D is not colliding with adventurer collision.**

A: The Adventurer was using `set_position()` directly for movement instead of setting velocity for `move_and_slide()`. When you teleport a `CharacterBody2D` via `set_position`, Godot's physics engine doesn't track it as normal physics movement, so `Area2D` body overlap detection doesn't fire. Fix: use `set_velocity()` then let `move_and_slide()` handle the actual movement.

---

**Q: When I push against the slime it gets stuck along with my movement.**

A: Now that the Adventurer uses `move_and_slide()`, it physically collides with the Slime's `CharacterBody2D`. They need to be on separate collision layers so they don't block each other, while the `Sight` area can still detect the Adventurer.
- Adventurer: layer 2, mask 1 (collides with world, not enemies)
- Slime body: layer 4, mask 1 (collides with world, not player)
- Slime Sight area: mask 2 (detects player layer)

---

**Q: Why can the slime be pushed down but not up?**

A: `CharacterBody2D` defaults to `GROUNDED` motion mode, which treats movement along `up_direction` (Vector2::UP, negative Y) asymmetrically — downward collisions are "floor" contacts, upward collisions are "ceiling" contacts. For a top-down game use `FLOATING` mode (`motion_mode = 1` in the scene file), which treats all directions uniformly.

---

**Q: Slime needs to keep track of target/player. Should I wrap Rc around Gd<Node2D>?**

A: No. `Gd<T>` is already reference-counted internally in godot-rust — cloning it just increments the ref count. Store it directly as `Option<Gd<Node2D>>`. Be aware: if the target is freed while the Slime holds a reference, calling methods on it will panic. Guard with `target.is_instance_valid()` before use.

---

**Q: What's the `#[func]` macro for?**

A: It exposes the Rust method to Godot's scripting/signal system. Without it, Godot doesn't know the method exists — you can't connect signals to it, call it from GDScript, or reference it with `Callable::from_object_method`. It's required for any method used as a signal callback.

---

**Q: How do I cast the body in a `body_entered` callback to a Slime instance?**

A: Use `try_cast()` or `cast()`:
```rust
if let Ok(slime) = body.try_cast::<Slime>() {
    slime.bind_mut().take_damage(10);
}
```
- `try_cast()` returns `Result<Gd<Slime>, Gd<Node2D>>` — safe, won't panic
- `cast()` returns `Gd<Slime>` directly but panics if the cast fails
- `into()` won't work for downcasting — it's for going up the hierarchy (e.g. `Gd<Slime>` → `Gd<Node>`)

---

**Q: For different enemy types should I use enums?**

A: Depends on the use case:
- **Enum** — good when enemy differences are small/known upfront, compiler forces exhaustive handling
- **`try_cast` per type** — idiomatic in godot-rust, cast to each type in sequence
- **Duck typing via `call()`** — most flexible, define a shared `#[func]` like `take_damage` on all enemies and call it generically:
```rust
body.call("take_damage", &[10_i32.to_variant()]);
```
Any node with a `take_damage` `#[func]` will respond, no cast needed.

---

**Q: Got `Gd<Slime>` — how do I get the Slime Rust struct out of it?**

A: Use `bind()` for immutable or `bind_mut()` for mutable access:
```rust
let s = slime.bind();         // Ref<Slime>
let mut s = slime.bind_mut(); // RefMut<Slime>
s.take_damage(10);
```
It's a guard similar to `RefCell::borrow()` — holds a borrow lock while in scope.

---

**Q: `bind_mut()` doesn't work, got `GdMut`.**

A: `GdMut<Slime>` is the guard type returned by `bind_mut()`. It implements `DerefMut` so you can call methods directly on it:
```rust
slime.bind_mut().take_damage(10);
```

---

**Q: Got `no field 'take_damage' on type GdMut<'_, Slime>` but the method exists.**

A: The method needs to be `pub` to be callable from another module:
```rust
pub fn take_damage(&mut self, damage: u32) { ... }
```

---

**Q: Why does the tutorial use `_process` for slime instead of RigidBody2D and physics?**

A: `CharacterBody2D` + manual movement gives exact control over enemy AI behavior. `RigidBody2D` hands control to the physics engine — you'd fight it to get predictable movement (velocity overridden by forces, friction, mass, etc.). For enemies that intentionally move toward a target, `CharacterBody2D` is almost always the right pick. `RigidBody2D` makes sense for physically simulated objects like boulders or projectiles.

**Q: Tween example for slime knockback?**

A: Tweening `position` directly conflicts with `move_and_slide()`. Prefer a knockback velocity that decays in `physics_process`:
```rust
pub fn apply_knockback(&mut self, direction: Vector2) {
    self.knockback = direction * 300.0;
}

fn physics_process(&mut self, _delta: f64) {
    self.knockback = self.knockback.lerp(Vector2::ZERO, 0.2);
    let movement = if self.knockback.length() > 5.0 {
        self.knockback
    } else {
        self.velocity
    };
    self.base_mut().set_velocity(movement);
    self.base_mut().move_and_slide();
}
```
If you do use a tween, store the upcast first and pass by reference:
```rust
let obj: Gd<Object> = self.to_gd().upcast::<Object>();
let mut tween = self.base_mut().create_tween().unwrap();
tween.tween_property(&obj, "position".into(), new_pos.to_variant(), 0.15);
```

---

**Q: Got type mismatch resolving `<Gd<Object> as ToGodot>::Pass == ByValue`.**

A: In newer gdext, `tween_property` expects `&Gd<Object>` not `Gd<Object>` by value. Store the upcast in a variable and pass a reference:
```rust
let obj: Gd<Object> = self.to_gd().upcast::<Object>();
tween.tween_property(&obj, ...);
```

---

**Q: Cloning the base — is base a smart pointer?**

A: No. `base()` returns `GdRef<T>`, not a `Gd<T>`. Use `self.to_gd()` to get a proper `Gd<Self>` reference to the node itself, then upcast:
```rust
let obj: Gd<Object> = self.to_gd().upcast::<Object>();
```

---

**Q: `to_gd()` docs warn about double borrow panic — example?**

A: The panic happens if you call `bind()`/`bind_mut()` on the result while `&mut self` is still active:
```rust
// PANICS
fn some_method(&mut self) {
    let gd = self.to_gd();
    gd.bind_mut(); // double borrow — self already mutably borrowed
}
```
Safe usage: only pass the result into an engine call, never call `bind`/`bind_mut` on it within the same method:
```rust
// SAFE - obj passed to engine, never re-borrowed in Rust
let obj: Gd<Object> = self.to_gd().upcast::<Object>();
tween.tween_property(&obj, ...);
```

**Q: How do I use the enums for `set_ease` and `set_trans`?**

A:
```rust
use godot::classes::tween::{EaseType, TransitionType};

tweener
    .set_trans(TransitionType::EXPO)
    .set_ease(EaseType::OUT);
```
Common combos for knockback:
- `EXPO` + `OUT` — fast burst, quick stop
- `BACK` + `OUT` — slight overshoot
- `QUAD` + `OUT` — smooth deceleration

**Q: How do I get the size from a `Sprite2D` in healthbar.rs?**

A:
```rust
let size = sprite.get_texture().unwrap().get_size(); // Vector2i
self.default_width = size.x;
self.default_height = size.y;
```
Use `sprite.get_rect().size` instead if you need the size accounting for scale (returns `Vector2` in local space).

---

**Q: How do I update the region rect inside `update_health`?**

A: Use `set_region_enabled` + `set_region_rect` to crop the sprite width based on current health:
```rust
pub fn update_health(&mut self, current: i32, max: i32) {
    if let Some(sprite) = &mut self.health_sprite_2d {
        let ratio = current as f32 / max as f32;
        let new_width = (self.default_width as f32 * ratio) as i32;
        sprite.set_region_enabled(true);
        sprite.set_region_rect(Rect2 {
            position: Vector2::ZERO,
            size: Vector2::new(new_width as f32, self.default_height as f32),
        });
    }
}
```

---

**Q: Why do I need `bind_mut()` for `update_health` — is it because it's a custom Rust class?**

A: Yes. `bind_mut()` is needed for any `Gd<T>` where `T` is your own Rust/GDExtension class. Built-in Godot classes (`Gd<Sprite2D>`, `Gd<Area2D>`) call methods directly via FFI into C++. Your own classes need `bind()`/`bind_mut()` to safely access the Rust struct behind the `Gd` pointer.

---

**Q: Is `bind_mut()` from godot-rust?**

A: Yes. `bind()`/`bind_mut()` are methods on `Gd<T>` defined by gdext. They're the equivalent of `RefCell::borrow()`/`borrow_mut()` — providing safe access to the inner Rust struct with runtime borrow checking.

---

**Q: Why don't built-in classes need `bind_mut()`?**

A: Because built-in Godot classes don't have a Rust struct behind them — they live entirely in Godot's C++ engine. godot-rust generates thin wrappers that call directly into the engine via FFI. `bind_mut()` exists specifically to safely access Rust-side fields. There's nothing like that for `Sprite2D` — calling `sprite.set_region_rect(...)` just passes straight through to C++.

```
Gd<Sprite2D>  ->  FFI call  ->  C++ Sprite2D object
                  (no Rust struct to borrow)

Gd<HealthBar> ->  borrow guard  ->  Rust HealthBar struct  ->  Godot object
                  (bind_mut needed to safely access Rust fields)
```
